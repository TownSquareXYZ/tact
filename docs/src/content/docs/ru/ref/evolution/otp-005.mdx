Это предложение определяет способ адресации контрактов по их аргументам вместо их исходных данных.

## Мотивация

Исходные данные могут сильно отличаться от аргументов. Это позволяет нам избежать выполнения ненадежного кода из другого контракта в контексте текущего или выполнения кода TVM off-chain для развертывания, что в некоторых случаях может быть рискованно.

## Спецификация

Эта спецификация определяет способ записи аргументов в ячейку данных инициализации для чтения кодом контракта во время развертывания.

### Префикс

Префикс определяется самим смарт-контрактом, но по умолчанию он предполагается как `один нулевой бит`. Префикс используется кодом контракта для различия между развернутым и неразвернутым состоянием.

### Кодирование аргументов

Аргументы кодируются с помощью [автоматического кодировщика](/ru/ref/evolution/otp-004).

### Требования к контракту

- Контракт ДОЛЖЕН предоставлять get-метод `lazy_deployment_completed`, который возвращает `true`, если контракт развернут, и `false` в противном случае.
- Контракт ДОЛЖЕН предоставлять интерфейс `org.ton.deploy.lazy.v0`.

## Недостатки

- Контракты могут находиться в полуразвернутом состоянии
- Существует несколько способов записи аргументов, которые могут привести к другим данным инициализации и другому адресу
- Вы можете развернуть предварительно инициализированный контракт, и он будет иметь другой адрес, оставаясь при этом полностью функциональным
- Непредсказуемое потребление газа при развертывании. Обычно развертывание обходится дорого, но это предложение делает его еще дороже.
